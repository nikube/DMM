# DMM Module Uninstall — Specification

**Version:** 1.0.0-draft
**Date:** 2026-03-27
**Related:** [DMM Specification](DMM_SPECIFICATION.md)

---

## 1. Overview

Dolibarr has no mechanism to fully uninstall a module. Disabling a module only removes database metadata (menus, permissions, constants) but leaves files in `/custom/`. DMM adds a proper uninstall feature that handles both DB cleanup and file deletion, with a backup safety net.

---

## 2. Current State

| Action | DB metadata | Files in /custom/ | DMM registry |
|---|---|---|---|
| Dolibarr disable | Removed | Untouched | N/A |
| DMM "Remove" (dashboard) | Untouched | Untouched | Deleted |
| Manual `rm -rf` | Untouched | Deleted | Untouched |

**Problem:** No single action performs a complete uninstall. Old module files accumulate indefinitely.

---

## 3. Proposed Uninstall Flow

### User Action

On the module detail page (`admin/module.php`), an "Uninstall" button is available for installed modules.

### Steps

```
1. User clicks "Uninstall"
2. Confirmation popup:
   "This will disable {module} in Dolibarr, delete its files from /custom/,
    and create a backup first. Continue?"
3. On confirm:
   a. Pre-flight: check write permissions on /custom/{module_id}/
   b. Backup: create backup via DMMClient::createBackup()
   c. Disable: call module descriptor's remove() method
      → cleans menus, permissions, constants, boxes, cron jobs
   d. Delete files: remove /custom/{module_id}/ directory
   e. Update registry: set installed = 0, installed_version = NULL in llx_dmm_module
   f. Show success message with note that backup is available for rollback
```

### What Gets Cleaned

| Item | Cleaned by | Method |
|---|---|---|
| Menu entries | Dolibarr `remove()` | `delete_menus()` |
| Permissions | Dolibarr `remove()` | `delete_permissions()` |
| Constants | Dolibarr `remove()` | `delete_const()` |
| Boxes/widgets | Dolibarr `remove()` | `delete_boxes()` |
| Cron jobs | Dolibarr `remove()` | `delete_cronjobs()` |
| Module files | DMM | `dol_delete_dir_recursive()` or `recursiveCopy` pattern |
| Module activation constant | Dolibarr `remove()` | `_unactive()` |
| DMM registry | DMM | Update `installed = 0` |

### What is NOT Cleaned

| Item | Reason |
|---|---|
| Module's own database tables | Convention: Dolibarr never drops tables on disable. Data preservation. |
| Extra fields created by module | Same convention. Could be re-used on reinstall. |
| Documents generated by module | In `DOL_DATA_ROOT`, not in `/custom/`. User data. |
| DMM registry entry | Kept with `installed = 0` so re-install is easy (repo + token preserved). |
| DMM backups | Kept for rollback. Subject to retention policy cleanup. |

---

## 4. Safety Measures

### Backup Before Delete

A full backup of the module directory is created before any deletion. Stored in `DOL_DATA_ROOT/dolimodulemanager/backups/`. Can be restored from the Backups section in DMM Settings.

### Self-Protection

DMM cannot uninstall itself (`module_id === 'dolimodulemanager'`). The uninstall button is hidden for DMM's own entry.

### Core Module Protection

Core Dolibarr modules cannot be uninstalled (existing `dmm_is_core_module()` check).

### Permission Check

Pre-flight write permission check on the module directory before attempting deletion (existing `checkWritePermissions()` method).

### Confirmation Required

Double confirmation: user must click "Uninstall" then confirm in a popup dialog.

---

## 5. UI

### Module Detail Page (`admin/module.php`)

New "Uninstall" button in the action bar, styled as `butActionDelete` (red). Only visible when:
- Module is installed (`$mod->installed == 1`)
- Module is not DMM itself
- User has `dolimodulemanager->write` permission

```
[Check now]  [Update v1.2.0]  [Uninstall]
                                  (red)
```

### Confirmation Popup

Standard Dolibarr `formconfirm`:
- Title: "Uninstall module"
- Message: "This will disable {module_id} in Dolibarr and delete its files from /custom/{module_id}/. A backup will be created first. This does NOT delete the module's database tables or documents."
- Actions: Yes / No

### Success Message

"Module {module_id} uninstalled. Backup saved — you can restore it from Settings > Backups."

---

## 6. Dashboard Impact

### "Remove" Action Renamed

The current "Remove" action on the dashboard (trash icon) is renamed to "Unregister" to clarify:
- **Unregister** = remove from DMM tracking only (no files deleted, no Dolibarr changes)
- **Uninstall** = full removal (disable + delete files + backup)

Confirmation message updated: "Remove this module from DMM tracking? The module files and Dolibarr activation are NOT affected."

## 7. Implementation Notes

### Calling Dolibarr's remove()

To disable a module programmatically:

```php
// Find and load the module descriptor
$descriptorFile = findDescriptor(DOL_DOCUMENT_ROOT.'/custom/'.$module_id);
$className = basename($descriptorFile, '.class.php');
include_once $descriptorFile;
$modInstance = new $className($db);
$modInstance->remove();
```

This is safe — `remove()` only deletes metadata, never tables or files.

### File Deletion

Use DMM's existing `recursiveCopy` pattern in reverse, or `dol_delete_dir_recursive()` from Dolibarr. Both handle nested directories.

### Registry Update

After uninstall, the DMM registry row is kept but updated:
```sql
UPDATE llx_dmm_module SET installed = 0, installed_version = NULL WHERE module_id = '{id}'
```

This preserves the `github_repo` and `fk_dmm_token` so the module can be reinstalled without re-configuration.

## 8. Edge Cases

| Scenario | Behavior |
|---|---|
| Module is active in Dolibarr but not in DMM registry | Cannot uninstall via DMM (no entry). User must disable manually in Dolibarr. |
| Module files already deleted manually | `remove()` still runs (cleans DB). File deletion step is a no-op. Registry updated. |
| Uninstall fails mid-way (e.g., permission error on some files) | Backup exists. Module may be in partial state. DMM shows error with backup restore instructions. |
| Module has dependencies (other modules depend on it) | Dolibarr's `remove()` handles this — it checks `requiredby` and refuses to disable if dependents are active. DMM should check the return value and abort if < 0. |

---

## 9. Not in Scope (v1)

- **Drop module tables**: Too dangerous. No convention in Dolibarr for this. Could destroy user data.
- **Uninstall from dashboard list**: Only available from module detail page (requires explicit navigation).
- **Batch uninstall**: One module at a time.
- **Auto-uninstall on hub removal**: Removing a hub does not uninstall its modules.
